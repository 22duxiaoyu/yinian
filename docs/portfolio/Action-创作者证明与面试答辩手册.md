## 3. 你实际做出的关键决策

### 决策 1：从“记录工具”改成“思考到行动的闭环”

**最初问题：**单纯记录网页过于简单，无法体现 AI 产品经理能力。  
**你的判断：**作品集需要展示产品定义、人机协作、数据闭环和 AI 落地，而不只是 UI。  
**结果：**形成“捕捉 → 理解 → 确认 → 行动 → 反馈”的核心链路。

### 决策 2：AI 不直接替用户下结论

**风险：**AI 根据少量个人记录给出强结论，会显得武断且不可信。  
**你的判断：**洞察必须有原始依据、置信度和“确认/驳回”，决定权属于用户。  
**结果：**洞察页面把“需要你确认”提升为最高权重，并允许查看依据后再转行动。

### 决策 3：工作台必须体现“今天要做什么”

**问题：**早期工作台像不可点击的数据 Banner，信息很多但没有工作入口。  
**你的判断：**工作台应该聚焦今日行动、待确认判断和最近输入，减少重复信息。  
**结果：**页面从展示型仪表盘转为任务型入口。

### 决策 4：灵感库统一手动记录和导入材料

**问题：**导入文件与手动灵感分成两套视觉结构，中间出现大面积空白，层级反向。  
**你的判断：**两类内容都是 AI 的事实来源，应从搜索区下方连续堆叠；点击时像抽卡一样展开。  
**结果：**统一为等尺寸的层叠卡片，并支持 PDF/DOCX 解析与“沉淀为灵感”。

### 决策 5：周报不是固定海报，而是动态决策地图

**问题：**固定一张图既像展示稿，也不能适配本周内容多少。  
**你的判断：**周报应该说明输入、判断、行动和下一步，并根据信息量改变结构。  
**结果：**形成 sparse / balanced / dense 三种布局，由输入量和 AI 共同选择。

### 决策 6：动效必须服务于状态反馈

**问题：**宠物、弹簧抽卡和错误对号等动效会分散注意，甚至让状态更难理解。  
**你的判断：**保留处理、成功、完成和展开收回等必要动效，删除炫技或不自然的效果。  
**结果：**动效主要用于告诉用户“正在处理、已保存、已完成、已返回”。

### 决策 7：从本地 Demo 升级为真实云端产品

**问题：**LocalStorage 只能在同一浏览器使用，无法让朋友跨设备登录。  
**你的判断：**作品集需要真实账户、数据隔离、跨设备同步、私有文件和服务端 AI。  
**结果：**接入 Supabase 全栈能力，并把 DeepSeek 密钥放在服务端。

## 5. 你必须真正理解的内容

### 产品层

你需要不看稿也能解释：

1. 为什么 Action 不是普通笔记和待办工具；
2. 目标用户现在只是待验证假设，下一步怎么验证；
3. 为什么核心指标不是记录数量，而是洞察进入行动；
4. 为什么洞察一定要有依据、置信度和人工确认；
5. 工作台、灵感库、洞察、行动、周报各自只解决什么；
6. 为什么当前不做团队协作、自动提醒和复杂项目管理。

### AI 层

你需要讲清：

1. 模型在三个节点发挥作用：洞察、对话、周报；
2. Prompt 输入了哪些用户数据，输出 JSON 长什么样；
3. 什么是幻觉、错误引用和过度推断；
4. 为什么要服务端调用 DeepSeek，而不是把 Key 放浏览器；
5. 如何评测引用准确率、洞察相关性和行动可执行性；
6. 模型失败、结构错误、空输入时产品如何降级。

### 技术与数据层

你不必假装是后端工程师，但需要理解：

1. Supabase Auth 负责账户和会话；
2. Postgres 存 notes、documents、insights、weekly_reports；
3. RLS 通过 `auth.uid()` 保证用户只能访问自己的数据；
4. Storage 私有桶保存原文件，路径按用户 ID 隔离；
5. Realtime 用来让其他设备收到数据变化；
6. Edge Functions 在服务端读取数据、调用 DeepSeek 并写回结果；
7. GitHub Pages 只托管静态前端，不保存数据库密码和模型密钥。

## 6. 三分钟演示脚本

### 0:00-0:30 问题

“很多人不是没有记录，而是记录完之后没有形成判断和下一步。Action 想解决的是从零散输入到真实行动之间的断层。”

### 0:30-1:10 输入

1. 打开灵感库；
2. 快速输入一条观察；
3. 展示 Word/PDF 导入和层叠卡片；
4. 强调原始表达不会被 AI 覆盖。

### 1:10-2:00 洞察

1. 点击“用 AI 做洞察”；
2. 展示全局分析反馈；
3. 打开第一条洞察的原始依据；
4. 说明“AI 提议，人做决定”；
5. 确认洞察并转成行动。

### 2:00-2:35 行动

1. 进入行动页；
2. 展示行动来源；
3. 标记完成；
4. 说明完成结果会进入下一轮判断。

### 2:35-3:00 周报

1. 打开动态周报；
2. 指出输入、核心判断、行动结果和下周实验；
3. 总结：“它不是帮用户写一篇周报，而是保留一条可以被验证的个人决策链。”

## 7. 十分钟项目讲解结构

1. **问题与用户：1 分钟**  
   为什么记录、聊天、待办分别不能独立解决这个问题。

2. **核心闭环：1 分钟**  
   输入、洞察、确认、行动、反馈之间的关系。

3. **关键体验：3 分钟**  
   演示灵感库、洞察证据、行动完成和动态周报。

4. **AI 设计：2 分钟**  
   Prompt 结构、人机边界、JSON 校验、空状态和评测方案。

5. **技术落地：1 分钟**  
   Supabase、RLS、私有存储、Realtime、Edge Function、DeepSeek。

6. **取舍与反思：1 分钟**  
   为什么删除宠物和无效动效，为什么不做复杂团队功能。

7. **验证计划：1 分钟**  
   承认尚无真实留存，用目标用户测试、埋点和 AI 回归集推进下一阶段。

## 8. 高频面试追问

### “这个产品和 Notion AI、备忘录或 ChatGPT 有什么区别？”

Action 的差异不在“也能记录、也能聊天”，而在固定的证据闭环：多条个人输入形成洞察，洞察要经过用户确认，再转成行动，行动结果回到下一轮分析。它把 AI 从通用回答者变成一个受约束的个人判断辅助层。

### “为什么用户不直接把笔记复制给 ChatGPT？”

一次复制可以得到一次答案，但缺少持续上下文、原始证据连接、确认状态、行动结果和周报历史。Action 的价值来自长期结构，而不是单次模型回答。这个差异仍需要通过真实留存验证。

### “AI 的置信度可信吗？”

当前置信度是模型生成并经过 0-100 边界校验的辅助信息，不是统计学概率。它只能用于帮助用户比较判断，不能作为真实性证明。下一阶段应使用标注集校准置信度与实际正确率的关系。

### “如何避免 AI 幻觉？”

限制模型只读取当前用户数据；Prompt 要求区分事实、推断和待验证；输出必须包含 evidence_refs；用户能查看原文并确认或驳回；结构错误和空输入不写入结果。还需要用人工标注集持续测错误引用率。

### “为什么周报要由 AI 选择布局？”

固定模板会在数据少时显得空，在数据多时隐藏重要证据。系统先按输入量给出 sparse、balanced、dense 建议，再让 AI 结合主题复杂度选择，但最终只允许三个受控值，避免模型随意生成前端结构。

### “你在项目中最大的产品取舍是什么？”

从“加更多功能和动效”转向“让行为链更清楚”。删除了 AI 宠物、不可点击 Banner 和不自然弹簧动效，把视觉权重集中到待确认判断和当前行动。这个取舍说明高级感不等于元素更多，而是用户更清楚下一步。

### “为什么选 Supabase？”

对个人项目而言，它能在短周期内提供 Auth、Postgres、RLS、Storage、Realtime 和 Edge Functions，减少自建后端成本，同时仍保留清晰的数据和权限边界。代价是依赖平台、邮件和函数配额，规模化前需要补充运维方案。

### “为什么用 AI coding agent？你自己的贡献在哪里？”

AI coding agent 负责加速实现和修改，我负责提出问题、定义闭环、选择方案、持续验收并决定删改。能够讲清需求、架构、风险、指标和下一步验证，并能用 Git 提交和实际产品证明迭代过程，就是这个项目中产品负责人的核心贡献。
